Review the ongoing use of IT tools and change approach as needed

I have no complaints to make about the software used to actually write the code required (Visual Studio Code), it did exactly what was needed of it and more, with the autocomplete feature in particular proving exceptionally useful.

On that point, GitHub also works as expected, though the method of keeping it updated is somewhat arcane, and I needed to keep looking it up every time I pushed an update to it after not doing so for a few hours.

While I can see the use Trello has for projects of substantial scope, or those completed by a group, in a solo project this narrow/short, it didn’t really save any more time or make work more efficient than it would have been without it as far as I can tell. This project was a very linear step to step process without much to forget or that could potentially fall by the wayside, and no coordination with others was required, making it not particularly useful.

Describe whether the IT tools selected were appropriate

Again, Visual Studio Code was the highlight here, and is easily better than similar software I’ve used to write HTML in the past (Though admittedly it isn’t that hard to be better than Notepad or Notepad++)

As mentioned earlier, Trello isn’t particularly useful in a task of this scale and complexity, taking more time to set up and maintain than it saves in productivity. I didn’t think GitHub was going to be particularly useful for this task either, until I had to work on the project from home for a day, during which having access to the repository became very useful. Assess the strengths and weaknesses of your final project

I believe that splitting up the content of the blog into multiple pages was a good idea, otherwise there’s a very real chance that the blog could have become one homogenous wall of text that would be very hard to fix with styling.

On the subject, I am pleased with the aesthetics. Despite being relatively simple I like to think it has a sort of minimalistic style to it now, with the dark background contrasting the light text highlighting the words on the page, and the fixed navbar making transition from one page to the others simple.

With regards to weaknesses, the same style I referred to just now does make it seem a bit dated, like it’s a website from the mid 2000’s. Additionally, the pages are too similar, looking almost identical with little to split them up aside from the content of the text, and despite being split up the wall of text problem hasn’t been solved completely, just reduced.

Describe further improvements you can make to your project

When I was styling the website, I wanted to make the Navbar ‘sticky’, so when you scrolled down the page title would be left behind, just showing the linkbar and saving a small amount of screenspace, while fitting the websites style. Unfortunately when I was trying to figure out how to do this, it would have required introducing Javascript into the style and I simply didn’t have the time to figure out how everything worked for implementing this into the website.

To help solve the wall of text problem, I was interested in the idea of splitting each section of text into a drop-down bar, where the user could click a section and it would expand downwards, showing the relevant text, and they could click on the title again to reduce it, similar to the FAQ section of some websites. If I had more time to work on the project, this is likely one of the avenues I would take next.

Describe how you would go about testing digital solutions

There’s a couple methods to use to go about testing digital projects, that are relevant to us right now, at least. The first is to put your project into all the conditions it is expected to work, and observe what happens. For example, does your website function when zoomed in or zoomed out? Does it work on a mobile device? What about this different mobile device? And so on, until you have found any errors that occur in normal usage, so you can proceed to fix them. In this project, that means testing it in different browsers, Chrome and Safari, for now.

The second method is called User Testing. Essentially, you show the website to people who will then see if they can navigate it, find what they want, if everything is legible and fit for their purposes. This is a good way to find out if the human elements of the website are in order, if buttons and links are where people expect them to be, or if the style looks decent.

Typically both methods are employed together when testing any digital solution that has a human element. (IE: If it needs to be used by humans.)
